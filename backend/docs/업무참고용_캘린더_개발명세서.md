## 1. 개요

### 1.1 기능 정보

| 항목 | 값 |
|------|-----|
| **기능 명칭** | 업무 참고용 캘린더 |
| **메뉴 경로** | 메인 상단 내비게이션 `캘린더` (`/calendar`) |
| **용도** | PC / MOBILE 관련 주요 일정(이벤트, 이슈, 작업 등)을 월 단위로 공유·조회 |

### 1.2 목적

- **운영·모니터링 업무 참고용**으로, 게임별 일정/이벤트를 한눈에 볼 수 있는 달력 UI 제공
- **KST(한국 표준시)** 기준으로 날짜/시간을 처리하여, 운영 기준 시간과 정확히 일치하도록 함
- **권한 분리**:
  - ADMIN / LEAD / SUPERADMIN: 이벤트 등록·수정·삭제 가능
  - AGENT / VIEWER: 조회 전용 (이벤트 상세 보기만 가능)

### 1.3 관련 파일

| 구분 | 경로 |
|------|------|
| 프론트엔드 페이지 | `src/Calendar.tsx` |
| 인증 컨텍스트 | `src/auth/AuthContext.tsx` |
| 백엔드 라우트 | `backend/routes/calendar.routes.js` |
| 백엔드 컨트롤러 | `backend/controllers/calendar.controller.js` |
| 인증 미들웨어 | `backend/middlewares/auth.middleware.js` |
| DB 스키마 | `backend/libs/schema.sql` (`CalendarEvent` 테이블) |
| DB 마이그레이션 | `backend/migrations/004_add_calendar_event.sql` |

## 2. UI / 동작 흐름

### 2.1 화면 구성

1. **상단 영역**
   - 좌측: 제목 `업무 참고용 캘린더`
   - 우측:
     - **구분 필터 드롭다운** (`전체` / `PC` / `MO`)
     - (관리자 전용) **이벤트 등록 버튼**
2. **본문 영역**
   - 두 개의 카드(Card)로 구분:
     - 카드 1: **현재 월** 캘린더
     - 카드 2: **다음 월** 캘린더
   - 각 카드 내부:
     - 요일 헤더(일~토)
     - 날짜 셀(grid) + 해당 날짜에 포함되는 이벤트 목록
3. **모달 다이얼로그**
   - **이벤트 등록/수정 모달** (ADMIN/LEAD/SUPERADMIN 전용)
   - **이벤트 상세 보기 모달** (모든 권한에서 사용, AGENT/VIEWER는 읽기 전용)

### 2.2 입력 필드 정의

1. **이벤트 등록/수정 모달**
   - **구분(platform) \***:
     - UI: `Select` 컴포넌트
     - 값: `'PC' | 'MO'`
   - **시작 날짜(startDate) \***:
     - UI: `<input type="date">`
     - 형식: `YYYY-MM-DD` (KST 기준)
     - 범위: **현재 월 1일 ~ 다음 월 마지막 일** 사이
   - **시작 시간(startTime) \***:
     - UI: `<input type="time">`
     - 형식: `HH:mm`
   - **종료 날짜(endDate) \***:
     - UI: `<input type="date">`
     - 형식: `YYYY-MM-DD` (KST 기준)
     - 범위: **현재 월 1일 ~ 다음 월 마지막 일** 사이
   - **종료 시간(endTime) \***:
     - UI: `<input type="time">`
   - **제목(title) \***:
     - UI: `<input type="text">`
     - 필수, 빈 문자열 불가(트림 후 체크)
   - **링크(link)**:
     - UI: `<input type="url">`
     - 선택값, 빈 문자열 허용 → 저장 시 `null` 처리

2. **이벤트 상세 보기 모달**
   - 구분 (PC/MO)
   - 시작 일시 (KST, `toLocaleString('ko-KR')`)
   - 종료 일시 (KST)
   - 링크 (있을 경우 클릭 가능한 URL, 새 탭)
   - (관리자 전용) 수정 / 삭제 버튼

### 2.3 구분별 필터 동작

- 상단 **구분 필터** 상태: `platformFilter: 'ALL' | 'PC' | 'MO'`
- 필터 값에 따른 동작:
  - `'ALL'`: 모든 이벤트 표시
  - `'PC'`: `event.platform === 'PC'` 인 이벤트만 표시
  - `'MO'`: `event.platform === 'MO'` 인 이벤트만 표시
- 필터는 **프론트엔드에서만 적용**되며, 서버로부터는 요청 기간 내의 모든 이벤트를 받아온 뒤 화면 단에서 필터링 한다.

## 3. 권한 정책

### 3.1 역할 정의 (요약)

- `ADMIN`, `LEAD`, `SUPERADMIN`: 관리자 권한 (이하 **관리자**)
- `AGENT`, `VIEWER` 등 그 외: 일반 사용자 (이하 **에이전트**)

역할 정보는 JWT 페이로드의 `role` 필드에 포함되며, `AuthContext` 에서 `user.role` 로 접근한다.

### 3.2 백엔드 권한 제어

1. **미들웨어**
   - `authenticate(req, res, next)`  
     - Authorization 헤더의 Bearer 토큰을 검증  
     - 성공 시 `req.user = { id, email, role, name }` 설정  
     - 실패 시 `401 Unauthorized` 반환
   - `requireRole(allowedRoles: string[])`
     - `req.user.role` 을 대문자로 변환
     - `allowedRoles` 배열(대문자 변환) 중 하나에 포함되지 않으면
       - 로그 경고 후 `403 Forbidden` 응답 (`Insufficient permissions`)

2. **캘린더 라우트 권한**

`backend/routes/calendar.routes.js` 기준:

- `GET /api/calendar/events`
  - 미들웨어: `authenticate`
  - **역할 제한 없음** (로그인한 유저면 모두 조회 가능)
- `POST /api/calendar/events`
  - 미들웨어: `authenticate`, `requireRole(['ADMIN', 'LEAD'])`
- `PUT /api/calendar/events/:id`
  - 미들웨어: `authenticate`, `requireRole(['ADMIN', 'LEAD'])`
- `DELETE /api/calendar/events/:id`
  - 미들웨어: `authenticate`, `requireRole(['ADMIN', 'LEAD'])`

### 3.3 프론트엔드 권한 제어

1. **권한 판별 로직**
   - `const { token, user } = useAuth();`
   - `const isAdmin = user?.role === 'ADMIN' || user?.role === 'LEAD' || user?.role === 'SUPERADMIN';`

2. **UI 제어**
   - 이벤트 등록 버튼:
     - `isAdmin === true` 일 때만 노출
   - 캘린더 셀 내 이벤트 박스 클릭:
     - `isAdmin === true`: **편집 모달** 열기 (`handleEditEvent` + 폼 세팅)
     - `isAdmin === false`: **상세 보기 모달**만 열고 수정 불가
   - 상세 보기 모달 내 버튼:
     - `isAdmin === true`: `수정`, `삭제`, `닫기`
     - `isAdmin === false`: `닫기`만 노출

3. **보호 로직 (추가 방어선)**
   - `handleSaveEvent`:
     - `if (!token || !isAdmin) { alert('권한이 없습니다. 관리자만 이벤트를 등록/수정할 수 있습니다.'); return; }`
   - `handleDeleteEvent`:
     - `if (!token || !isAdmin) { alert('권한이 없습니다. 관리자만 이벤트를 삭제할 수 있습니다.'); return; }`
   - 서버 응답이 `403` 인 경우:
     - 사용자 친화적인 메시지 `"권한이 없습니다. 관리자만 이벤트를 등록/수정(또는 삭제)할 수 있습니다."` 표시

## 4. DB 설계

### 4.1 테이블 정의 — `CalendarEvent`

`backend/libs/schema.sql` 및 `backend/migrations/004_add_calendar_event.sql` 기준:

```sql
CREATE TABLE IF NOT EXISTS CalendarEvent (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  platform TEXT NOT NULL,           -- 'PC' 또는 'MO'
  startDate DATETIME NOT NULL,      -- ISO/SQLite DATETIME 문자열 (KST 기준 논리)
  endDate DATETIME NOT NULL,
  title TEXT NOT NULL,
  link TEXT,
  createdAt DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updatedAt DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX IF NOT EXISTS idx_calendarevent_platform ON CalendarEvent(platform);
CREATE INDEX IF NOT EXISTS idx_calendarevent_startDate ON CalendarEvent(startDate);
CREATE INDEX IF NOT EXISTS idx_calendarevent_endDate ON CalendarEvent(endDate);
CREATE INDEX IF NOT EXISTS idx_calendarevent_startDate_endDate ON CalendarEvent(startDate, endDate);
```

### 4.2 필드 의미

- `platform`:
  - `'PC'` 또는 `'MO'`
  - UI의 "구분" 필드와 1:1 매핑
- `startDate`, `endDate`:
  - 문자열 형식: `YYYY-MM-DDTHH:mm:ss` (타임존 정보 없음)
  - KST 기준 시각을 그대로 기록 (로컬에서 KST 시각을 생성한 뒤 문자열로 변환)
- `title`:
  - 이벤트 제목
- `link`:
  - 관련 문서/공지/웹페이지 URL (nullable)
- `createdAt`, `updatedAt`:
  - SQLite `datetime('now')` 사용

## 5. API 명세

### 5.1 공통 사항

- URL prefix: `/api/calendar`
- 인증:
  - 모든 엔드포인트에 Authorization 헤더 필요
  - 형식: `Authorization: Bearer {JWT_TOKEN}`

### 5.2 이벤트 목록 조회 — GET `/api/calendar/events`

- **권한**: 로그인한 모든 사용자 (ADMIN/LEAD/AGENT/VIEWER)
- **쿼리 파라미터**
  - `startDate` (선택): `YYYY-MM-DD HH:mm:ss`
    - 조건: `endDate >= startDate`
  - `endDate` (선택): `YYYY-MM-DD HH:mm:ss`
    - 조건: `startDate <= endDate`
- **동작**
  - 기본 SQL:
    - `SELECT * FROM CalendarEvent WHERE 1=1`
  - `startDate` 제공 시:
    - `AND endDate >= ?`
  - `endDate` 제공 시:
    - `AND startDate <= ?`
  - 정렬: `ORDER BY startDate ASC`
- **응답 형식 (성공)**
  - `200 OK`
  - Body:
    - `data: { events: CalendarEvent[] }`
    - `message: "Events retrieved successfully"`

### 5.3 이벤트 생성 — POST `/api/calendar/events`

- **권한**: ADMIN, LEAD (백엔드에서 `requireRole`로 제한)
- **Request Body (JSON)**
  - `platform`: `"PC"` 또는 `"MO"` (필수)
  - `startDate`: `YYYY-MM-DDTHH:mm:ss` (필수)
  - `endDate`: `YYYY-MM-DDTHH:mm:ss` (필수)
  - `title`: 문자열 (필수, trim 후 빈값 불가)
  - `link`: 문자열 또는 `null` (선택)
- **유효성 검증**
  - `platform` 누락 또는 값이 `PC/MO` 가 아니면 오류
  - `startDate` 또는 `endDate` 누락 시 오류
  - `title`이 비어 있으면 오류
  - `new Date(startDate) > new Date(endDate)` 인 경우 오류
- **성공 응답**
  - `201 Created`
  - `data: { event }` (방금 생성된 레코드 조회 결과)
  - `message: "Event created successfully"`

### 5.4 이벤트 수정 — PUT `/api/calendar/events/:id`

- **권한**: ADMIN, LEAD
- **Request Params**
  - `id`: 수정 대상 이벤트 ID
- **Request Body (JSON)**
  - 허용 필드: `platform`, `startDate`, `endDate`, `title`, `link` (부분 업데이트 가능)
- **동작**
  1. `id` 존재 확인 (`SELECT * FROM CalendarEvent WHERE id = ?`)
     - 없으면 `404 Not Found`
  2. Request Body 에 있는 필드만 `SET` 절에 포함
  3. 항상 `updatedAt = datetime('now')` 로 갱신
  4. 수정 후 `SELECT` 로 최신 레코드 반환
- **유효성 검증**
  - `platform` 이 존재할 경우 값 검증 (`PC` 또는 `MO`)
  - `startDate`/`endDate` 둘 다 존재할 경우, `startDate <= endDate` 확인

### 5.5 이벤트 삭제 — DELETE `/api/calendar/events/:id`

- **권한**: ADMIN, LEAD
- **Request Params**
  - `id`: 삭제 대상 이벤트 ID
- **동작**
  1. `id` 존재 여부 확인
  2. 없으면 `404 Not Found`
  3. 있으면 `DELETE FROM CalendarEvent WHERE id = ?`
  4. 성공 시 `data: {}` 와 `"Event deleted successfully"` 메시지 반환

## 6. 프론트엔드 동작 (상세)

### 6.1 상태 관리

`src/Calendar.tsx` 기준 주요 상태:

- `currentMonth: Date`
- `events: CalendarEvent[]`
- `loading: boolean`
- `selectedEvent: CalendarEvent | null`
- `isDialogOpen: boolean` (등록/수정 모달)
- `platformFilter: 'ALL' | 'PC' | 'MO'`
- `formData` (등록/수정 폼)
  - `platform: 'PC' | 'MO'`
  - `startDate: string` (`YYYY-MM-DD`)
  - `startTime: string` (`HH:mm`)
  - `endDate: string`
  - `endTime: string`
  - `title: string`
  - `link: string`

### 6.2 KST(한국 표준시) 처리

1. **KST 기준 Date 생성 헬퍼**
   - `getKSTDate(year, month, day, hour = 0, minute = 0, second = 0)`
   - 브라우저 로컬 타임존을 이용하되, 논리적으로는 KST 기준으로 사용

2. **월 범위 계산 (`getMonthRange`)**
   - 기준: 현재 시각 `now`
   - `start`:
     - `getKSTDate(currentYear, currentMonthIndex, 1, 0, 0, 0)` → 현재 월 1일 00:00:00
   - `end`:
     - `getKSTDate(currentYear, currentMonthIndex + 2, 0, 23, 59, 59)`  
       (다음 월 마지막 일 23:59:59)
   - 반환: `{ start, end }`

3. **이벤트 로딩 시 사용**
   - `start`/`end`를 `YYYY-MM-DD HH:mm:ss` 문자열로 변환하여 API 쿼리로 전달:
     - 예: `2026-02-01 00:00:00`, `2026-03-31 23:59:59`

4. **입력 필드 `min` / `max`**
   - `startDate`/`endDate`의 `min`/`max` 속성에 `formatKSTDate(getMonthRange().start/end)` 적용  
     → **현재 월 + 다음 월** 범위 밖의 날짜는 UI 상에서 선택 불가

5. **이벤트 범위 체크**
   - 등록/수정 시:
     - `eventStart = parseKSTDateTime(formData.startDate, formData.startTime)`
     - `eventEnd = parseKSTDateTime(formData.endDate, formData.endTime)`
   - 유효성:
     - `eventStart <= eventEnd`
     - `getMonthRange().start <= eventStart`
     - `eventEnd <= getMonthRange().end`

### 6.3 이벤트 목록 조회 (`loadEvents`)

1. `if (!token) return;`
2. `getMonthRange()`로 `start`, `end` 계산
3. 문자열 포맷: `YYYY-MM-DD HH:mm:ss`
4. Fetch:
   - `GET /api/calendar/events?startDate={start}&endDate={end}`
   - 헤더: `Authorization: Bearer ${token}`
5. `res.ok` 가 아니면 에러 처리 (`alert('이벤트를 불러오는데 실패했습니다.')`)
6. 성공 시: `setEvents(body.data?.events || [])`

### 6.4 이벤트 등록/수정 (`handleSaveEvent`)

1. 권한 체크: `!token || !isAdmin` 이면 얼럿 후 종료
2. 필수 필드 검증:
   - `startDate`, `startTime`, `endDate`, `endTime`, `title.trim()`
3. 날짜/시간 파싱:
   - `parseKSTDateTime(YYYY-MM-DD, HH:mm)` → `Date`
4. 시각 비교 및 범위 검증 (6.2 참고)
5. ISO 문자열 변환:
   - `YYYY-MM-DDTHH:mm:ss` 형식으로 변환 (타임존 없음)
6. API 호출:
   - 새 이벤트: `POST /api/calendar/events`
   - 수정: `PUT /api/calendar/events/{id}`
7. 응답 처리:
   - 403: `"권한이 없습니다. 관리자만 이벤트를 등록/수정할 수 있습니다."` 메시지
   - 기타: 서버 `message` 사용 또는 `"Failed to save event"`
8. 성공 시:
   - 모달 닫기
   - 폼 리셋
   - `loadEvents()` 재호출

### 6.5 이벤트 삭제 (`handleDeleteEvent`)

1. 권한 체크: `!token || !isAdmin` → 얼럿 후 종료
2. 사용자 확인: `confirm('이 이벤트를 삭제하시겠습니까?')` → 취소 시 종료
3. API 호출:
   - `DELETE /api/calendar/events/{id}`
4. 응답 처리:
   - 403: `"권한이 없습니다. 관리자만 이벤트를 삭제할 수 있습니다."`
   - 기타 실패: `"이벤트 삭제에 실패했습니다."`
5. 성공 시:
   - `loadEvents()` 재호출

### 6.6 날짜별 이벤트 그룹화

1. **플랫폼 필터 적용**
   - `filteredEvents = platformFilter === 'ALL' ? events : events.filter(e => e.platform === platformFilter);`
2. **그룹화 로직**
   - 각 `event` 에 대해:
     - `start = new Date(event.startDate)`
     - `end = new Date(event.endDate)`
     - `current = start` 를 날짜 단위로 증가시키며 `endDateOnly` 까지 반복
     - 각 날짜에 대해:
       - `dateKey = formatKSTDate(current)` → `YYYY-MM-DD`
       - `eventsByDate[dateKey]` 배열에 이벤트 추가

### 6.7 캘린더 렌더링

1. **월 그리드 계산 (`renderMonthGrid`)**
   - 입력: `(year, monthIndex)`
   - 1일 ~ 마지막 일까지 `Date` 배열 생성
   - 첫 주: `firstDay.getDay()` 만큼 앞에 placeholder 날짜(`new Date(0)`) 추가
   - 이후 7일 단위로 잘라 2차원 배열 `weeks: Date[][]` 생성
   - 마지막 주도 길이가 7이 될 때까지 placeholder로 채움

2. **오늘 표시**
   - `today = new Date()`
   - `todayKey = formatKSTDate(today)`
   - 셀 날짜 `day` 에 대해 `dateKey === todayKey` 이면 배경/텍스트 색상 강조

3. **이벤트 표시**
   - `dayEvents = eventsByDate[dateKey] || []`
   - 각 이벤트에 대해:
     - 색상:
       - PC: 파랑 계열 (`bg-blue-*`, `text-blue-*`)
       - MO: 초록 계열 (`bg-green-*`, `text-green-*`)
     - 제목: 한 줄로 truncate
     - 시간: `startDate`, `endDate` 를 `toLocaleTimeString('ko-KR', { hour: '2-digit', minute: '2-digit' })` 로 표현
     - 클릭:
       - 관리자: 편집 모달
       - 에이전트: 상세 보기 모달

## 7. 오류 / 예외 처리

| 상황 | 처리 |
|------|------|
| 토큰 없음 | 백엔드 `401 Unauthorized`, 프론트엔드는 캘린더 진입 전 로그인이 필요하다는 전제 |
| 권한 부족 (등록/수정/삭제) | 백엔드 `403 Forbidden`, 프론트 `"권한이 없습니다. 관리자만 이벤트를 등록/수정/삭제할 수 있습니다."` 표시 |
| 날짜/시간 필드 누락 | 클라이언트에서 얼럿 `"모든 필수 필드를 입력해주세요."` |
| 시작 > 종료 | `"종료 날짜/시간은 시작 날짜/시간보다 이후여야 합니다."` |
| 월 범위 외 날짜 | `"이벤트는 현재월과 다음월 범위 내에서만 등록할 수 있습니다."` |
| 서버 DB 에러 | 백엔드에서 로그 기록 후 `"Failed to retrieve/create/update/delete event"` 메시지와 함께 500 응답 |

## 8. 재현성 체크리스트

- [ ] 로그인 후 `/calendar` 진입 시, 현재월·다음월 2개의 캘린더 카드가 표시된다.
- [ ] 상단 구분 필터에서 `전체` / `PC` / `MO` 를 선택하면, 해당 플랫폼의 이벤트만 표시된다.
- [ ] ADMIN/LEAD/SUPERADMIN 계정에서만 **이벤트 등록** 버튼이 보이고, 이벤트 박스를 클릭하면 편집 모달이 뜬다.
- [ ] AGENT/VIEWER 계정에서는 이벤트 등록 버튼이 보이지 않고, 이벤트 클릭 시 읽기 전용 상세 모달만 열린다.
- [ ] 이벤트 등록 시, 현재월 ~ 다음월 범위를 벗어나거나 종료가 시작보다 이전이면 클라이언트에서 경고 후 저장되지 않는다.
- [ ] 동일한 기간에 다일 다일 간 걸치는 이벤트는 시작일부터 종료일까지 모든 날짜 셀에 표시된다.
- [ ] DB에 직접 INSERT 한 이벤트도 `GET /api/calendar/events` 로 정상 조회되며, 쿼리의 `startDate`/`endDate` 필터가 의도대로 동작한다.
- [ ] 관리자 계정에서 이벤트 생성/수정/삭제 시, 캘린더가 새로고침 없이 최신 상태로 갱신된다.
